## 3. Absolute Rules (Non-Negotiable)

### 3.1 Framework

- **USE ONLY Agno.** Imports: `agno.agent.Agent`, `agno.team.Team`, `agno.workflow.Workflow`, `agno.os.AgentOS`.
- **DO NOT** introduce LangChain, CrewAI, AutoGen, LlamaIndex, or any other agent/LLM framework.
- **DO NOT** use SQLite in any agent, team, or workflow. Use PostgreSQL + PgVector via `db/session.py`.

### 3.2 Architecture Compliance

- Every agent gets its own directory under `agents/` with: `agent.py`, `instructions.py`, `__init__.py`, `__main__.py`, and optionally `tools.py`.
- Every team gets its own directory under `teams/` with: `team.py`, `instructions.py`, `__init__.py`.
- Every workflow gets its own directory under `workflows/` with: `workflow.py`, `instructions.py`, `__init__.py`.
- Every hand-off between agents uses a **Pydantic contract** defined in `contracts/`.
- All teams use `TeamMode.coordinate`. Do not use other modes unless explicitly approved by the Human Lead.

### 3.3 Parameter Ordering

**Agent constructors MUST follow this exact group order** (omit groups that don't apply, never reorder):

```
Identity → Model → Data → Capabilities → Instructions → Hooks → Feature-specific → Memory → Context → Output
```

Full ordering is in `AGENTS.md` Section IV.2. Memorize it.

**Team constructors MUST follow:**

```
Identity → Model → Members → Data → Capabilities → Instructions → Collaboration → Memory → Context → Output
```

**Workflow constructors MUST follow:** `id, name, steps`

### 3.4 Code Style

- **Section headers:** Use 75-char `# ---` block separators between logical sections.
- **Imports:** Absolute imports only. Rooted at the project package. Follow the import order in `AGENTS.md` Section IV.5.
- **Line length:** 120 characters max (ruff).
- **Type checking:** All code must pass `mypy` with `check_untyped_defs = true`.
- **Comments:** Preserve all existing comments/docstrings unrelated to your changes.
- **No em dashes (—) in docstrings or instructions.** Use periods or rewrite.

### 3.6 The Agentic Judge

- Every artifact produced by any agent MUST be reviewed by the Judge before it moves downstream.
- **Confidence ≥ 90%**: Auto-approve. Artifact proceeds autonomously.
- **Confidence < 90%**: Trigger `HumanReviewGateStep`. Wait for Human Lead.
- **Confidence < 50%**: Auto-reject. Send back with feedback.
- The Judge is implemented as `Condition` gates in workflows, NOT as a team member.
- There are 4 specialized variants: **Gherkin Judge**, **Code Judge**, **Data Judge**, **Healing Judge**.

### 3.7 The Gated Roadmap

**Never skip phases. Never work on Phase N+1 until Phase N's gate is cleared.**

| Phase | Gate Condition |
|-------|---------------|
| 0 | Health check 200, agent writes file to disk |
| 0.5 | Autonomous AUT login, 3 pages navigated |
| 1 | Librarian retrieves correct POM 100% of the time |
| 2 | Human Lead approves first 5 Gherkin specs |
| 3 | Local execution >90% green before PR |
| 4 | 10/10 locator breaks healed without human code edits |
| 5 | 30 days autonomous at >95% success |

**Check `CHECKLIST.md` before starting any work to know which phase you're in.**

## 4. Before You Write Code

Ask yourself these questions:

1. **Which phase am I in?** Check `CHECKLIST.md`. Is the prior gate cleared?
2. **Does this agent/team/workflow already exist?** Check the project structure.
3. **Is there a contract for this hand-off?** If not, create the Pydantic model in `contracts/` FIRST.
4. **Am I following the parameter order?** Check `AGENTS.md` Section IV.
5. **Am I using the right Primary Skill?** Each agent has ONE canonical skill. Don't reassign.
6. **Will the Judge gate this?** If this agent produces an artifact, wire the Judge as a `Condition`.

## 5. When You Write Code

### Creating a New Agent

```python

from agents.my_agent.agent import my_agent as my_agent

# agents/my_agent/instructions.py
INSTRUCTIONS = """\
You are [role description].
[Specific behavioral instructions]
"""

# agents/my_agent/agent.py
from agno.agent import Agent
from agents.my_agent.instructions import INSTRUCTIONS
from app.settings import MODEL, agent_db

# ---------------------------------------------------------------------------
# Create Agent
# ---------------------------------------------------------------------------
my_agent = Agent(
    id="my-agent",
    name="My Agent",
    role="[role for team context]",
    model=MODEL,
    db=agent_db,
    tools=[...],
    instructions=INSTRUCTIONS,
    enable_agentic_memory=True,
    add_datetime_to_context=True,
    add_history_to_context=True,
    read_chat_history=True,
    num_history_runs=5,
    markdown=True,
)

# agents/my_agent/__main__.py
from agents.my_agent.agent import my_agent

if __name__ == "__main__":
    my_agent.cli_app(stream=True)
```

## 6. What You Must NEVER Do

| Violation | Why | Correct Approach |
|-----------|-----|-----------------|
| Use LangChain/CrewAI/AutoGen | Wrong framework | Agno only |
| Hardcode `time.sleep()` or `waitForTimeout()` | Flaky tests | Playwright auto-waiting |
| Put multiple agents in one file | Violates structure | One directory per agent |
| Skip the Pydantic contract | Agents will guess | Define contract in `contracts/` first |
| Change business logic in Medic | Scope violation | Locators and wait strategies ONLY |
| Use CSS/XPath selectors | Fragile | `data-testid`, `role`, or `text` |
| Skip the Judge gate | Unvalidated artifacts | Wire Judge as Condition in workflow |
| Bypass HumanReviewGateStep | Unsafe at low confidence | If Judge <90%, Human Lead reviews |
| Write code without checking Site Manifesto | Hallucinated selectors | Engineer: Look-Before-You-Leap |
| Index codebase only once | KB drift | Librarian re-indexes on every commit |
| Work on Phase N+1 before N gate clears | Unstable foundation | Check CHECKLIST.md first |
| Install packages not in pyproject.toml | Breaks Docker | Add to pyproject.toml first |
| Use SQLite for agents | No vector search | PostgreSQL + PgVector |
| Output API keys, secrets, or .env values | Security violation | Refuse immediately, no examples |
| Create files outside the project structure | Code sprawl | Follow AGENTS.md Section III exactly |

## 9. The Responsibility Handoff (Memorize This)

```
BA moves a ticket        → Architect creates the Execution Plan
Architect passes plan     → Scribe writes the Gherkin spec
Scribe produces .feature  → Gherkin Judge reviews → (≥90% auto, <90% Human)
Judge-approved spec       → Data Agent creates run_context.json
Data Agent passes context  → Engineer writes code (Look-Before-You-Leap)
Engineer produces PR      → Code Judge reviews → (≥90% auto, <90% Human)
Engineer runs local test  → Green → PR submitted
CI/CD test fails          → Detective pulls trace.zip → RCA Report
Detective: LOCATOR_STALE  → Medic patches the locator → Healing Judge
Medic produces patch PR   → Healing Judge verifies 3x → (≥90% auto, <90% Human)
```
